# Security posture

> **v1 — internal use only, no external network exposure.**
>
> This release is designed to run on a private network behind your existing
> perimeter (corporate LAN, dedicated subnet, jump-host, or a VPN). It does
> **not** ship user authentication, transport security, or RBAC. Treat the
> entire dashboard as a single-tenant, fully-privileged surface available to
> anyone who can route to it. If you need TLS, multi-user, or
> auditable-per-operator access today, terminate that in front of the
> dashboard with your own ingress (nginx + auth, an authenticating reverse
> proxy, an oauth2-proxy sidecar, etc.).

## Threat model summary

| Attribute | v1 |
| --- | --- |
| Operator authentication | **None.** Anyone reaching the web/api can read everything and assign any config to any onboarded VM. |
| Transport encryption | Plain HTTP between browser and web container, web ↔ api, and api ↔ Postgres. Use a TLS-terminating reverse proxy or service-mesh sidecar in front. |
| Agent authentication | Per-agent SHA-256-hashed bearer key issued at registration. Required on every `/api/agents/:agentId/*` call. |
| Inter-service authentication | None inside the trust boundary. The api connects to Postgres with a static password. |
| Secrets at rest | Service-principal credentials in env vars (compose) or `Secret` (k8s). Postgres data on the bundled volume / PVC, unencrypted at rest unless your underlying disk is encrypted. |
| Audit | Append-only `audit_events` table with `actor_type` of `ui`, `agent`, or `system`. |
| Multi-tenant | Not supported. |

## No user authentication

Every `/api/*` UI route is unauthenticated. There is no session cookie, no
login page, no per-user RBAC. This is **deliberate** for v1 to keep the
template's getting-started loop short. The corollary:

- Treat the URL of the dashboard as a credential. Don't put it on the
  internet, or in a shared bookmarks list, or in a wiki page.
- All UI mutations are stamped in `audit_events` as `actor_type = 'ui'` with
  no `actor_id`. There is no way to attribute an action to an individual
  operator.
- The Jobs and Audit Events tabs surface the full payload of every action,
  including the names and Azure IDs of every server. Anyone with UI access
  can read those.

## Per-agent API keys

The agent-facing endpoints (`/api/agents/:agentId/*` except `/register`) are
the **only** routes that require authentication, and they use bearer tokens.

- Plaintext key returned **once**, at registration
  ([`POST /api/agents/register`](../apps/api/src/routes/agents.ts)). Never
  read back from the database.
- Stored as `sha256(plaintext)` in `agent_keys.key_hash`
  ([`apps/api/src/lib/agentAuth.ts`](../apps/api/src/lib/agentAuth.ts) and
  [`apps/api/src/lib/tokens.ts`](../apps/api/src/lib/tokens.ts)).
- Comparison uses Node's `timingSafeEqual` to avoid string-compare timing
  leaks.
- Multiple non-revoked rows per server are allowed. To rotate without
  downtime: `POST /api/servers/:id/rotate-key` issues a new key, leaves the
  old one usable until you redeploy the agent, then your operational process
  revokes the old one (currently a manual SQL update —
  `UPDATE agent_keys SET revoked_at = now() WHERE id = ...`).
- The current rotation flow assumes the operator drives the agent-side
  reconfiguration out-of-band (e.g. a fresh
  `Register-DashboardAgent.ps1 -ProvisionToken <new>` invocation). A self-service
  UI rotation flow is on the v2 list.

## Provision tokens

Distinct from long-lived agent keys:

- Generated by [`POST /api/servers/:id/provision-token`](../apps/api/src/routes/servers.ts)
  via `crypto.randomBytes(32)` (`base64url`).
- Stored only on the corresponding `provision` job's `payload.token` —
  there is no separate `provision_tokens` table.
- Lifetime defaults to `AZURE_RUNCOMMAND_TIMEOUT_MINUTES` (30 minutes).
  After expiry the token is rejected by `POST /api/agents/register` with
  `401 provision token expired` and the scheduler eventually fails the
  pending `provision` job with `errorCode = 'TokenExpired'`.
- Single-use: when a register call succeeds, the api flips the job to
  `success` and mints a long-lived agent key. Replaying the same token
  against `register` afterwards returns `401 invalid provision token`
  because the job is no longer in `queued|running|failed`.

## Azure service principal

The api authenticates to Azure using
[`DefaultAzureCredential`](https://learn.microsoft.com/azure/developer/javascript/sdk/credential-chains#use-defaultazurecredential-for-flexibility).
Configure **one** of:

- **Service principal env vars** (recommended for production):
  `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`. Set in
  `.env` (compose) or as a `Secret` referenced from `k8s/20-api-config.yaml`
  (k8s — keep it out of the `ConfigMap`).
- **`AzureCliCredential` fallback** for development: mount your
  `~/.azure` directory into the api container and the SDK will pick it up
  without env vars. Don't ship this to production.

The required RBAC on every target VM (or a containing scope) is the data
action `Microsoft.Compute/virtualMachines/runCommand/action`. Built-in role
that grants it: **Virtual Machine Contributor**. For least privilege, define
a custom role with only that single action and assign it on the VM scope.
The api's credential probe runs at boot and surfaces in `/healthz` as
`azure: ok | unconfigured | error`.

## Database backups contain config bodies

Every `config_revisions.yaml_body` is the **plain text** YAML the user saved.
That includes:

- Anything in `properties.*` of every resource.
- The full body of any `SetScript` / `TestScript` / `GetScript` for inline
  PowerShell.
- The full payload of every audit event (`audit_events.payload`) and every
  `dsc` run output (`run_results.dsc_output`).

**Do not put secrets, passwords, API tokens, or connection strings into config
YAML.** The Configs editor surfaces a heuristic warning if it detects
`password`, `secret`, `token`, or `credential` substrings, but it cannot
guarantee detection. Treat your Postgres backups as confidential to the
extent your most sensitive config is.

A KeyVault / external-secrets integration is on the v2 backlog.

## Audit log

`audit_events` is append-only at the schema level (no `UPDATE` or `DELETE`
issued from any code path) and is written:

- From the [`audit` Fastify plugin](../apps/api/src/plugins/audit.ts) for
  every UI mutation that returns 2xx — `actor_type = 'ui'`.
- Directly from agent endpoints (`actor_type = 'agent'`,
  `actor_id = agent:<uuid>`) for register / result / removal-ack /
  module-install.
- From the scheduler and job runners for system-driven transitions
  (`actor_type = 'system'`).

The UI exposes it on the **Audit** tab of each server's detail page and via
`GET /api/audit-events` with optional filtering by `entityType`, `entityId`,
`eventType`, or `actorType`. There is no built-in retention policy — a
periodic `DELETE FROM audit_events WHERE created_at < now() - interval '180 days'`
is a reasonable starting point if growth becomes a concern.

## Hardening backlog (what v2 should add)

- **Operator authentication.** Either a bearer-token for the UI
  (configurable, single shared token to start) or full OIDC integration via
  `oauth2-proxy` / Authelia / your IdP of choice.
- **TLS termination via ingress.** Today the
  [`k8s/40-ingress.yaml`](../k8s/40-ingress.yaml) is HTTP-only. Annotate it
  with cert-manager / a TLS secret and add an HTTPS redirect.
- **Network policies / firewalls.** Postgres and api containers should
  only accept traffic from the web container and from agent CIDRs
  respectively. `NetworkPolicy` resources are not shipped.
- **Sealed-secrets / external-secrets.** The current k8s manifests inline
  the SP credentials in a `Secret`. For a real cluster, source them from
  Azure Key Vault via the Secrets Store CSI driver or from `external-secrets`.
- **Scheduled DB backup job.** A `CronJob` running `pg_dump` to durable
  storage on a daily / hourly cadence. The lab restore loop is documented in
  [`k8s/README.md`](../k8s/README.md#persistent-storage), but there is no
  automation.
- **Rate limiting / abuse protection.** Fastify exposes a `rate-limit`
  plugin that should be wired in front of `/api/agents/register` and
  `/api/configs/parse` at minimum.
- **UI-driven agent-key rotation flow.** The api supports it
  (`/api/servers/:id/rotate-key`); the UI does not yet have the button.
- **Secret-store-backed config values.** A `${secret:keyVaultRef}` style
  templating layer in the YAML so secrets never enter the database.
